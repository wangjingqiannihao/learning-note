# 为什么容器 1 号进程是 systemd 而不是业务进程

## 核心结论

容器的 1 号进程不一定必须是业务进程。把 `systemd` 放在容器的 PID 1 位置，本质上是把容器当成一个更接近传统 Linux 系统的运行环境来管理，让 `systemd` 负责进程生命周期、信号转发、子进程回收、服务编排和系统初始化，业务进程则作为 `systemd` 管理的服务运行。

这种做法适合需要在单个容器内运行多个协作进程、复用传统 Linux 服务管理方式、或需要完整 init 语义的场景。如果容器只运行一个简单业务进程，并且业务进程能正确处理信号和子进程回收，直接让业务进程作为 1 号进程通常更简单。

## 容器 1 号进程的特殊性

在 Linux 中，PID 1 不是普通进程。它承担 init 进程的职责，主要包括接收容器运行时发送的停止信号、处理孤儿进程、回收僵尸进程，以及决定整个容器的生命周期。

容器运行时启动容器时，会把容器镜像中的入口命令作为容器内的 PID 1。这个进程一旦退出，容器通常也会退出。因此，PID 1 的行为会直接影响容器能否优雅停止、能否正确清理子进程，以及服务异常时是否能被正确拉起或退出。

## 业务进程直接作为 PID 1 的问题

很多业务程序是按照普通用户态进程设计的，并没有专门适配 PID 1 的语义。如果直接让业务进程作为容器 1 号进程，可能出现几个典型问题。

第一，信号处理不完整。容器停止时，运行时通常会向 PID 1 发送 `SIGTERM`。如果业务进程没有正确处理该信号，就可能无法执行优雅退出逻辑，例如停止接收新请求、完成已有请求、刷新缓存或关闭连接。

第二，子进程回收不可靠。如果业务进程会启动子进程，或者依赖脚本、插件、side task 等间接产生子进程，那么这些子进程退出后需要父进程调用 `wait()` 回收。普通业务进程如果没有实现这部分逻辑，容易留下僵尸进程。

第三，多个进程缺少统一管理。有些容器内不只一个进程，例如主服务、日志收集进程、定时任务、辅助守护进程等。如果业务进程作为 PID 1，它通常只关注自身逻辑，不能像 init 系统一样可靠地管理其他进程的启动顺序、依赖关系、重启策略和退出行为。

第四，传统服务脚本难以复用。部分软件依赖 `systemd` unit、`systemctl`、`tmpfiles`、`journald` 或类似系统级机制。如果没有 init 系统，这些组件需要额外改造，迁移成本会比较高。

## systemd 作为 PID 1 做了什么

当 `systemd` 作为容器内的 PID 1 时，它会承担 init 角色，业务服务则通过 unit 文件交给 `systemd` 管理。

典型结构如下：

```text
# 容器内进程关系示意
PID 1: systemd
  ├─ business.service    # 业务主进程，由 systemd 拉起和管理
  ├─ log-agent.service   # 可选：日志或辅助进程
  └─ timer.service       # 可选：定时任务或后台任务
```

在这种结构下，容器运行时只需要管理 `systemd` 这个顶层进程；容器内部的服务启动、停止、重启、依赖编排和状态检查则由 `systemd` 负责。

一个简化的 service 示例：

```ini
# /etc/systemd/system/business.service
[Unit]
# 服务描述，方便 systemctl status 查看
Description=Business Service

# 表示该服务在网络初始化后启动
After=network-online.target
Wants=network-online.target

[Service]
# 前台启动业务进程，不要让业务进程自己 daemonize
Type=simple

# 业务启动命令
ExecStart=/usr/local/bin/business-server --config /etc/business/config.yaml

# 收到停止命令时，systemd 会先向主进程发送 SIGTERM
KillSignal=SIGTERM

# 业务异常退出时自动重启
Restart=on-failure

# 两次重启之间等待 5 秒，避免异常时快速重启打满资源
RestartSec=5s

[Install]
# 开机进入 multi-user.target 时启动该服务
WantedBy=multi-user.target
```

## 这样做的好处

### 统一接管进程生命周期

`systemd` 能统一管理容器内所有服务的启动、停止、重启和状态。业务进程不需要自己实现复杂的 supervisor 逻辑，也不需要在一个入口脚本里手写大量进程管理代码。

当业务进程异常退出时，可以通过 `Restart=on-failure` 等策略自动拉起；当容器停止时，`systemd` 可以按 unit 关系有序停止服务，避免粗暴杀进程导致数据不一致或连接异常中断。

### 正确处理 PID 1 语义

`systemd` 天然就是 init 系统，能够处理 PID 1 的特殊职责，包括信号处理、孤儿进程接管和僵尸进程回收。这样可以避免普通业务进程不适配 PID 1 带来的隐藏问题。

这对会启动子进程的服务尤其重要。即使业务进程或脚本间接产生了子进程，`systemd` 也更适合作为顶层进程兜底管理。

### 支持多服务和依赖编排

有些容器不是严格的单进程模型，而是需要多个进程协同工作。例如业务进程启动前需要先准备目录、加载配置、启动辅助服务，或者需要定时任务和常驻进程同时存在。

`systemd` 可以通过 `After=`、`Before=`、`Requires=`、`Wants=` 等机制描述服务之间的依赖关系，比在 shell 脚本中串联启动命令更清晰，也更容易维护。

### 复用传统 Linux 运维能力

很多软件原本就是以 systemd unit 的方式部署的。容器内使用 `systemd` 后，可以复用已有的 service 文件、启动顺序、日志习惯和排障命令，例如：

```bash
# 查看业务服务状态
systemctl status business.service

# 查看业务服务日志
journalctl -u business.service

# 重启业务服务
systemctl restart business.service
```

这能降低传统服务容器化时的改造成本，尤其适合把虚拟机或物理机上的服务迁移到容器环境的场景。

### 让容器退出行为更可控

当 `systemd` 是 PID 1 时，容器是否退出不再完全取决于某一个业务进程的直接生命周期，而是可以通过 systemd target 和服务策略统一控制。

例如，业务进程异常退出后可以自动重启；如果关键服务持续失败，也可以让容器最终退出，由外部编排系统接管重建。这种模式比简单的 `while true` 脚本更明确，也更容易观察和排障。

## 需要注意的代价

使用 `systemd` 作为容器 PID 1 并不是所有场景的默认最优解。它会让容器更接近一个完整操作系统环境，因此也会带来更高的复杂度。

首先，镜像通常更大，启动链路更复杂。其次，运行 `systemd` 可能需要额外挂载 cgroup，并配置更高的容器权限。再次，容器内运行多个服务会弱化“一容器一主进程”的简单模型，排障边界也会变得更复杂。

因此，是否使用 `systemd` 取决于容器定位。如果容器只是运行一个无状态 Web 服务，业务进程自身也能正确处理 `SIGTERM` 和子进程，那么直接运行业务进程更轻量。如果容器承载的是传统系统服务、多进程组合、复杂启动依赖，或者需要完整 init 语义，那么让 `systemd` 作为 PID 1 更合适。

## 适用场景判断

可以优先考虑使用 `systemd` 作为 PID 1 的场景包括：容器内需要运行多个长期进程；服务之间存在明确启动顺序或依赖关系；需要复用已有 systemd unit；业务进程会派生子进程但自身没有可靠回收逻辑；需要统一的服务重启、日志查看和状态管理；容器更像轻量虚拟机或系统级运行环境。

不建议默认使用 `systemd` 的场景包括：容器只运行一个简单业务进程；业务进程已经正确处理停止信号和优雅退出；外部编排系统已经负责重启和健康检查；希望镜像尽量小、权限尽量低、运行模型尽量简单。

## 总结

容器 1 号进程使用 `systemd`，不是因为业务进程不能直接运行，而是因为某些场景下需要一个真正的 init 进程来管理容器内部的进程树和服务生命周期。

它的主要好处是：正确承担 PID 1 的特殊职责，统一管理多个服务，支持依赖编排和自动重启，复用传统 Linux 运维方式，并让容器的启动、停止和故障恢复更可控。

但这也意味着更高的复杂度和更重的运行环境。简单业务容器优先直接运行业务进程；复杂系统型容器或传统服务迁移场景，使用 `systemd` 作为 PID 1 会更稳妥。
